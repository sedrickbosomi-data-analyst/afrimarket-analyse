AfriMarket — Analyse Stratégique des Données¶

Contexte : AfriMarket est une entreprise e-commerce panafricaine (Électronique, Mode, Beauté, Maison) opérant dans plusieurs grandes villes d'Afrique francophone. La direction constate des variations de CA, un taux de retour préoccupant, des dépenses marketing élevées et des écarts de performance selon les villes.

Mission : produire une analyse stratégique complète (6 mois d'activité) permettant à la direction de prendre des décisions business.

Résumé de l'audit¶

Problème Détail Impact
Doublons stricts ~100 lignes strictement dupliquées Gonfle artificiellement le CA et les volumes
Villes mal orthographiées "Kinshassa" au lieu de "Kinshasa" Fausse l'analyse géographique (ville comptée 2x)
Catégories incohérentes "electronique" (minuscule, sans accent) vs "Électronique" Fausse l'analyse par catégorie
Statuts à la casse incohérente "Livrée" / "retournée" / "Annulée" Empêche les regroupements corrects (indicateur_retour, taux d'annulation)
Remises négatives 614 lignes, min -0.1 Erreur de saisie de signe — fausserait le chiffre d'affaires si non corrigé
Prix aberrants 632 lignes ≤ 0, min -50 Valeurs impossibles économiquement — traitées comme manquantes
Quantités nulles 608 lignes à 0 Ne représentent pas une vente réelle
Coûts marketing/livraison en texte Typage object au lieu de float Bloque tout calcul agrégé (ROI, coûts totaux)
Pas de coût produit Aucune colonne coût d'achat Marge brute non calculable directement → nécessite une hypothèse de taux de marge par catégorie

Conclusion de l'audit : le dataset est exploitable mais nécessite un nettoyage structuré avant toute analyse fiable. Aucun retraitement n'est fait sur les axes d'analyse (villes, catégories, canaux) sans le nettoyage préalable.

3. Feature Engineering¶

Hypothèse posée (documentée car aucun coût produit n'est fourni) : taux de marge brute par catégorie, basés sur des repères sectoriels usuels de l'e-commerce panafricain :

Catégorie Taux de marge brute supposé
Électronique 15 % (marge faible, produits à forte concurrence prix)
Mode 45 %
Beauté 55 %
Maison 35 %

Convention métier : une commande Annulée ne génère aucune vente réelle → son chiffre d'affaires est ramené à 0. Une commande Retournée a bien généré une vente (elle est comptée dans le CA et le profit) mais est signalée via indicateur_retour : c'est un signal de risque qualité, pas une perte de revenu immédiate.

In [20]:
TAUX_MARGE = {
    "Électronique": 0.15,
    "Mode": 0.45,
    "Beauté": 0.55,
    "Maison": 0.35,
}

# Chiffre d'affaires (0 pour les commandes annulées)
df["chiffre_affaires"] = np.where(
    df["statut_commande"] == "Annulée",
    0.0,
    df["prix_unitaire"] * df["quantite"] * (1 - df["remise"]),
)

# Marge brute estimée
df["marge_brute"] = df["chiffre_affaires"] * df["categorie"].map(TAUX_MARGE)

# Profit net estimé (marge brute - coûts logistiques et marketing de la commande)
df["profit_net"] = df["marge_brute"] - df["cout_livraison"] - df["cout_marketing"]

# Mois (période Y-M, pour les analyses d'évolution mensuelle)
df["mois"] = df["date_commande"].dt.to_period("M").astype(str)

# Indicateurs de statut
df["indicateur_retour"] = (df["statut_commande"] == "Retournée").astype(int)
df["indicateur_annulation"] = (df["statut_commande"] == "Annulée").astype(int)

# Nombre de commandes par client (sur les 6 mois observés)
df["nombre_commandes_par_client"] = df.groupby("id_client")["id_commande"].transform("count")

# Valeur vie client simplifiée = somme du CA généré par le client sur la période observée
df["valeur_vie_client"] = df.groupby("id_client")["chiffre_affaires"].transform("sum")
